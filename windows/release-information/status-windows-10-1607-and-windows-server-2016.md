<h3>February 2021</h3><table border ='0' class='has-text-wrap'>
<tr><td style='border-left-width:0px;border-right-width:0px;border-bottom-width:0px;font-size:1.5rem !important;margin-top:0px;margin-bottom:0px;' class='has-margin-top-none has-margin-bottom-none has-padding-left-none has-padding-right-none' colspan='3' width='100%'><div id='1560msgdesc'></div><h4 class='has-margin-top-none has-margin-bottom-none has-margin-right-none has-margin-left-none has-padding-left-none has-padding-right-none has-padding-top-none has-padding-bottom-none'>Windows Presentation Foundation (WPF) apps might close, or you may receive an error</h4></td></tr>

<tr><td width='100%' colspan='3'><div>After installing <a href='https://support.microsoft.com/help/4601318' target='_blank'> KB4601318</a>, Windows Presentation Foundation (WPF) apps might close, or you may receive an error similar to:</div><div class="ql-indent-1">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;Exception Info: System.NullReferenceException at System.Windows.Interop.HwndMouseInputProvider.HasCustomChrome(System.Windows.Interop.HwndSource, RECT ByRef)</div><div class="ql-indent-1">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;at System.Windows.Interop.HwndMouseInputProvider.GetEffectiveClientRect(IntPtr)</div><div class="ql-indent-1">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;at System.Windows.Interop.HwndMouseInputProvider.PossiblyDeactivate(IntPtr, Boolean)</div><div class="ql-indent-1">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;at System.Windows.Interop.HwndMouseInputProvider.Dispose()</div><div>&nbsp;</div><div>This occurs&nbsp;when disposing an HwndSource whose RootVisual is null, a situation that arises in Visual Studio when docking or splitting windows, and could arise in other apps.</div><div><br></div><div><strong>Affected platforms:</strong></div><ul><li>Client: Windows 10, version 1803; Windows 10 Enterprise LTSC 2016; Windows 10, version 1607</li><li>Server: Windows Server, version 1803; Windows Server 2016</li></ul><div></div><div><strong>Workaround: </strong>To mitigate this issue when using Microsoft Visual Studio, update to the latest version.&nbsp;This issue is resolved starting with the following releases:</div><ul><li><a href="https://docs.microsoft.com/visualstudio/releases/2019/release-notes-preview#16.9.0.pre.4.0" rel="noopener noreferrer" target="_blank" style="">Visual Studio 2019 version 16.9 Preview 4</a></li><li><a href="https://docs.microsoft.com/visualstudio/releases/2019/release-notes#16.8.5" rel="noopener noreferrer" target="_blank" style="">Visual Studio 2019 version 16.8.5</a></li><li><a href="https://docs.microsoft.com/visualstudio/releases/2019/release-notes-v16.7#16.7.12" rel="noopener noreferrer" target="_blank" style="">Visual Studio 2019 version 16.7.12</a></li><li><a href="https://docs.microsoft.com/visualstudio/releasenotes/vs2017-relnotes#15.9.33" rel="noopener noreferrer" target="_blank" style="">Visual Studio 2017 version 15.9.33</a></li></ul><div></div><div>Developers can mitigate this issue in their own apps by setting two AppContext switches using one of the methods described in <a href="https://docs.microsoft.com/dotnet/api/system.appcontext?view=netframework-4.8#appcontext-for-library-developers" rel="noopener noreferrer" target="_blank" style="">AppContext Class (System) under the heading “AppContext for library consumers”</a>.&nbsp;The switches are named <strong>Switch.System.Windows.Interop.MouseInput.OptOutOfMoveToChromedWindowFix</strong> and <strong>Switch.System.Windows.Interop.MouseInput.DoNotOptOutOfMoveToChromedWindowFix</strong>; both should be set to “true”.&nbsp;The first switch avoids the crash, but re-introduces the issue addressed in this update.&nbsp;The second switch is currently ignored, but will be recognized in a future .NET update that addresses the issue for the null-reference crash;&nbsp;it restores the behavior originally intended to be addressed in this update.</div><div><br></div><div>For example, using the app.config file method to apply the workaround at application scope:</div><div class="ql-indent-1">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&lt;AppContextSwitchOverrides value="Switch.System.Windows.Interop.MouseInput.OptOutOfMoveToChromedWindowFix=true; Switch.System.Windows.Interop.MouseInput.DoNotOptOutOfMoveToChromedWindowFix=true " /&gt;</div><div><br></div><div><strong>Next steps: </strong>We are working on a resolution and will provide an update in an upcoming release.</div><div align='right'><a href ='#known-issues'>Back to top</a></div></td></tr>
</table>

<tr><td width='100%' colspan='3'><div>After installing <a href="https://support.microsoft.com/help/4467684" target="_blank">KB4467684</a>, the cluster service may fail to start with the error "2245 (NERR_PasswordTooShort)" if the Group Policy "Minimum Password Length" is configured with greater than 14 characters.</div><div><br></div><div><strong>Affected platforms:</strong></div><ul><li>Client: Windows 10, version 1607; Windows 10 Enterprise LTSC 2016</li><li>Server: Windows Server 2016</li></ul><div></div><div><strong>Workaround:</strong> Set the domain default "Minimum Password Length" policy to less than or equal to 14 characters.</div><div><br></div><div><strong>Next steps:</strong> Microsoft is working on a resolution and will provide an update in an upcoming release.</div><div align='right'><a href ='#known-issues'>Back to top</a></div></td></tr>
