<tr><td width='100%' colspan='3'><div>After installing <a href='https://support.microsoft.com/help/5000802' target='_blank'> KB5000802</a>, you might receive an APC_INDEX_MISMATCH error with a blue screen when attempting to print to certain printers in some apps. This issue affects a subset of Type 3 printer drivers and does not affect printer drivers that are Type 4. If you are unsure which type your printer driver is, use the following steps:</div><ol><li>Press the Windows key+r or select <strong>Start </strong>and type <strong>run </strong>and select it.</li><li>Type <strong>printmanagement.msc </strong>and press enter or select the <strong>OK button</strong>.</li><li>If Print Management opens, continue to step 4. If you receive the error, <strong>Windows can not find 'printmanagement.msc'.</strong> then select <strong>Start </strong>and type <strong>manage optional features</strong> and select it. You will now need to start at step 1 again.</li><li>Expand <strong>Print Servers</strong>, expand your computer's name, and select <strong>Printers</strong>.</li><li>You can now see the Driver Type for each of your installed printer drivers.</li></ol><div><br></div><div><strong>Workaround: </strong>To mitigate the issue, follow the instructions in <a href="https://www.microsoft.com/videoplayer/embed/RWzqv8" rel="noopener noreferrer" target="_blank">this video</a>. The <strong>Command Prompt </strong>commands from the video are as follows (you will need to replace <strong>KX driver for Universal printing</strong> with the name of your printer, as explained in the video):</div><ul><li><strong>rundll32 printui.dll,PrintUIEntry /Xg /n “KX driver for Universal printing”</strong></li><li><strong>rundll32 printui.dll,PrintUIEntry /Xs /n "KX driver for Universal printing" attributes +direct</strong></li></ul><div></div><div><strong>Affected platforms:</strong></div><ul><li>Client: Windows 10, version 20H2; Windows 10, version 2004; Windows 10, version 1909; Windows 10, version 1809; Windows 10 Enterprise LTSC 2019; Windows 10, version 1803</li><li>Server: Windows Server, version 20H2; Windows Server, version 2004; Windows Server, version 1909; Windows Server, version 1809; Windows Server 2019; Windows Server, version 1803</li></ul><div></div><div><strong>Resolution:</strong>&nbsp;This issue was resolved in the out-of-band update&nbsp;<a href='https://support.microsoft.com/help/5001567' target='_blank'> KB5001567</a>. It is a cumulative update, so you do not need to apply any previous update before installing it. You will need to <strong>Check for updates </strong>to install <a href='https://support.microsoft.com/help/5001567' target='_blank'> KB5001567</a> and might need to also select <strong>Download and Install</strong>. For instructions on how to <strong>Check for updates</strong>, please see <a href="https://support.microsoft.com/windows/update-windows-10-3c5ae7fc-9fb6-9af1-1984-b5e0412c556a" rel="noopener noreferrer" target="_blank">Update Windows 10</a>. You can import this update&nbsp;into Windows Server Update Services (WSUS) manually. See the&nbsp;<a href="https://docs.microsoft.com/windows-server/administration/windows-server-update-services/manage/wsus-and-the-catalog-site#the-microsoft-update-catalog-site" rel="noopener noreferrer" target="_blank">Microsoft Update Catalog</a>&nbsp;for instructions.</div><div align='right'><a href ='#resolved-issues'>Back to top</a></div></td></tr>

intended for enforcement mode and will create issues in an environment where not all DCs are updated because it will explicitly reject certain types of non-compliant Kerberos&nbsp;tickets. <tr><td width='100%' colspan='3'><div>System and user certificates might be lost when updating a device&nbsp;from&nbsp;Windows 10, version 1809 or later&nbsp;to a later version of Windows 10.&nbsp;Devices will only&nbsp;be impacted&nbsp;if they have already&nbsp;installed&nbsp;any&nbsp;Latest cumulative&nbsp;update&nbsp;(LCU)&nbsp;released September 16, 2020 or later and&nbsp;then proceed to update to a later version of Windows 10 from&nbsp;media or an&nbsp;installation source&nbsp;which&nbsp;does not&nbsp;have an LCU released&nbsp;October 13, 2020&nbsp;or later&nbsp;integrated. This&nbsp;primarily happens when&nbsp;managed devices&nbsp;are updated&nbsp;using outdated bundles or media through an update management tool such as Windows Server Update Services (WSUS)&nbsp;or&nbsp;Microsoft Endpoint Configuration Manager.&nbsp;This might also happen when using outdated physical media or ISO images that do not have the latest updates integrated.&nbsp;</div><div>&nbsp;</div><div><strong>Note </strong>Devices using Windows Update for Business or that connect directly to Windows Update&nbsp;are not impacted.&nbsp;Any device connecting to Windows Update&nbsp;should always receive the latest versions of&nbsp;the feature update, including the latest&nbsp;LCU,&nbsp;without any extra steps.&nbsp;</div><div>&nbsp;</div><div><strong>Workaround:</strong> If you have already encountered this issue on your device, you can&nbsp;mitigate it&nbsp;within&nbsp;the uninstall window&nbsp;by&nbsp;going&nbsp;back to your previous version of Windows using the instructions&nbsp;<a href="https://support.microsoft.com/windows/recovery-options-in-windows-10-31ce2444-7de3-818c-d626-e3b5a3024da5#bkmk_section6" rel="noopener noreferrer" target="_blank">here</a>.&nbsp;The&nbsp;uninstall&nbsp;window might be 10 or 30 days depending on&nbsp;the configuration of&nbsp;your environment&nbsp;and the version you’re&nbsp;updating to.&nbsp;You will then need to update to the later version of Windows 10 after the issue is resolved in your environment.&nbsp;<strong>Note&nbsp;</strong>Within&nbsp;the uninstall window,&nbsp;you can increase the number of days you have to go back to your previous version of Windows 10 by using the DISM command&nbsp;/Set-OSUninstallWindow.&nbsp;You must make this change&nbsp;<strong>before</strong>&nbsp;the default&nbsp;uninstall&nbsp;window has lapsed.&nbsp;For&nbsp;more information,&nbsp;see&nbsp;<a href="https://docs.microsoft.com/windows-hardware/manufacture/desktop/dism-uninstallos-command-line-options" rel="noopener noreferrer" target="_blank">DISM&nbsp;operating system uninstall command-line options</a>.&nbsp;</div><div><br></div><div><strong>Affected platforms:</strong></div><ul><li>Client: Windows 10, version 20H2; Windows 10, version 2004; Windows 10, version 1909; Windows 10, version 1903</li><li>Server: Windows Server, version 20H2; Windows Server, version 2004; Windows Server, version 1909; Windows Server, version 1903</li></ul><div></div><div><strong>Resolution:</strong> This issue is now resolved&nbsp;when using the latest feature update bundles and refreshed media. Feature update bundles were released November 9, 2020 for Windows Server Update Services (WSUS) and Microsoft Endpoint Configuration Manager. Refreshed media was released November 3, 2020 on&nbsp;Volume Licensing Service Center (VLSC) and Visual Studio Subscriptions (VSS, formerly MSDN Subscriptions). For information on verifying you're using the refreshed media, see <a href="https://support.microsoft.com/help/4576779" rel="noopener noreferrer" target="_blank">How to address feature update refreshes in your environment</a>. If you are using or creating custom media, you will need to include an update released October 13, 2020 or later.</div><div><br></div><div><strong>Note</strong> If you are updating to Windows 10, version 20H2, this is only resolved with the feature update bundle released November 9, 2020. Refreshed media is not yet available on VLSC or VSS. Refreshed media for VLSC and VSS will be released in the coming weeks to address this issue and another&nbsp;<a href="https://docs.microsoft.com/windows/release-information/status-windows-10-20h2#1514msgdesc" rel="noopener noreferrer" target="_blank">known issue</a> that requires a media refresh. Please check the known issue <a href="https://docs.microsoft.com/windows/release-information/status-windows-10-20h2#1514msgdesc" rel="noopener noreferrer" target="_blank">here</a> for the status of the remaining Windows 10, version 20H2 known issue.</div><div align='right'><a href ='#resolved-issues'>Back to top</a></div></td></tr>

<tr><td width='100%' colspan='3'><div>When installing a third-party driver, you might receive the error,&nbsp;“Windows can’t verify the publisher of this driver software”. You may also see the error “No signature was present in the subject” when attempting to view the signature properties using Windows Explorer.</div><div><br></div><div>This issue occurs when one or more of the following is present in a driver package:</div><ul><li>an improperly formatted catalog file is identified during validation by Windows.&nbsp;Starting with this release, Windows will require the validity of DER encoded PKCS#7 content in catalog files. Catalogs files must be signed per section 11.6 of describing&nbsp;DER-encoding for SET OF members&nbsp;in <a href="https://www.itu.int/rec/T-REC-X.690/" rel="noopener noreferrer" target="_blank">X.690</a>.</li><li>a driver catalog file extension is not one of the supported extensions.</li></ul><div></div><div><strong>Affected platforms:</strong></div><ul><li>Client: Windows 10, version 20H2, Windows 10, version 2004; Windows 10, version 1909; Windows 10, version 1903; Windows 10, version 1809; Windows 10 Enterprise LTSC 2019; Windows 10, version 1803; Windows 10, version 1709; Windows 10 Enterprise LTSC 2016; Windows 10, version 1607; Windows 10 Enterprise 2015 LTSB; Windows 8.1</li><li>Server: Windows Server, version 20H2, Windows Server, version 2004; Windows Server, version 1909; Windows Server, version 1903; Windows Server, version 1809; Windows Server 2019; Windows Server, version 1803; Windows Server, version 1709 ; Windows Server 2016; Windows Server 2012 R2</li></ul><div></div><div><strong>Next steps: </strong>If this happens you should contact the driver vendor or device manufacturer (OEM) and ask them for an updated driver to correct the issue.</div><div align='right'><a href ='#resolved-issues'>Back to top</a></div></td></tr>

<tr><td width='100%' colspan='3'><div>After waking from sleep or hibernation, certain WWAN LTE modems might show no internet in the Network Connectivity Status Indicator (NCSI) in the notification area and might be unable to connect to the internet.</div><div><br></div><div>To safeguard your update experience, we have applied a compatibility hold on Windows 10 devices with affected WWAN LTE modems drivers installed from being offered Windows 10, version 2004 until the issue has been resolved. If your organization is using&nbsp;<a href="https://docs.microsoft.com/windows/deployment/update/update-compliance-monitor" rel="noopener noreferrer" target="_blank">Update Compliance</a>, the safeguard ID is 28428232.</div><div><br></div><div><strong>Workaround:</strong> To mitigate this issue if you are already on Windows 10, version 2004, you can enable then disable <strong>Airplane mode.</strong> To do this, you can select the <strong>Start button</strong>, type <strong>airplane mode</strong> and select it. In the settings dialog, toggle <strong>Airplane mode </strong>on then off again. You should now be able to connect as expected.</div><div><br></div><div><strong>Affected platforms:</strong></div><ul><li>Client: Windows 10, version 2004</li></ul><div></div><div><strong>Resolution: </strong>This issue was resolved in <a href='https://support.microsoft.com/help/4577063' target='_blank'> KB4577063</a>&nbsp;and the safeguard hold has been removed as of October 15, 2020.
