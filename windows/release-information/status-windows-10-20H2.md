<tr><td width='100%' colspan='3'><div>After installing <a href='https://support.microsoft.com/help/4586781' target='_blank'> KB4586781</a> on domain controllers (DCs) and read-only domain controllers (RODCs) in your environment, you might encounter Kerberos&nbsp;authentication issues. This is caused by an issue in how <a href="https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2020-17049" rel="noopener noreferrer" target="_blank">CVE-2020-17049</a> was addressed in these updates. As noted in <a href="https://msrc.microsoft.com/update-guide/en-US/vulnerability/CVE-2020-17049" rel="noopener noreferrer" target="_blank">CVE-2020-17049</a>, there are three registry setting values for PerformTicketSignature to control it, but in the current implementation you might encounter different issues with each setting:</div><ul><li>Setting the value to 0 might cause authentication issues when using S4U scenarios, such as scheduled tasks, clustering, and services for example line-of-business applications.</li><li>The default value setting of 1 might cause non-Windows clients authenticating to Windows Domains using Kerberos to experience authentication issues.</li><ul><li>With setting 1, clients attempting to renew a Kerberos&nbsp;ticket that should be renewable on a DC updated with <a href='https://support.microsoft.com/help/4586781' target='_blank'> KB4586781</a> will fail to renew the Kerberos&nbsp;ticket if it was issued from a DC that has not installed an update released November 11, 2020 or any DC running Windows Server 2008 R2 SP1 or Windows Server 2008 SP2.</li><li>Going from 0 to 1 might also cause this issue since there can be outstanding Kerberos tickets that are marked renewable, but will not be renewed by updated DCs.</li></ul><li>With the default value setting of 1, you might also have <a href="https://docs.microsoft.com/openspecs/windows_protocols/ms-kile/bac4dc69-352d-416c-a9f4-730b81ababb3" rel="noopener noreferrer" target="_blank">Cross realm referrals</a> failures on Windows and non-Windows devices for Kerberos referral tickets passing through domain DCs that have not installed an update released November 11, 2020 or any DC running Windows Server 2008 R2 SP1 or Windows Server 2008 SP2.&nbsp;This issue might happen if domain environment is partially updated or contains at least one Windows Server 2008 R2 SP1 or Windows Server 2008 SP2.</li><li>Setting the value to 2 is intended for enforcement mode and will create issues in an environment where not all DCs are updated because it will explicitly reject certain types of non-compliant Kerberos&nbsp;tickets. It should also not be used at this time if your environment contains DCs running Windows Server 2008 R2 SP1 or Windows Server 2008 SP2.</li></ul><div></div><div><strong>Note </strong>This issue only affects Windows Servers, Windows 10 devices and applications in enterprise environments.</div><div><br></div><div><strong>Affected platforms:</strong></div><ul><li>Server: Windows Server, version 20H2; Windows Server, version 2004; Windows Server, version 1909; Windows Server, version 1903; Windows Server, version 1809; Windows Server 2019; Windows Server 2016; Windows Server 2012 R2; Windows Server 2012</li></ul><div></div><div><strong>Next steps: </strong>We are working on a resolution and will provide an update as soon as more information is available.</div><div align='right'><a href ='#known-issues'>Back to top</a></div></td></tr>

The affected driver will be named <strong>Conexant HDAudio Driver</strong> under <strong>Sound, video and game controllers</strong> in <strong>Device Manager</strong> and have versions&nbsp;<strong>8.65.47.53​,&nbsp;8.65.56.51​, or 8.66.0.0 through 8.66.89.00 </strong>for<strong> chdrt64.sys </strong>or<strong> chdrt32.sys</strong>.</div><div><br></div><div>To safeguard your update experience, we have applied a compatibility hold on Windows 10 devices with affected Conexant or Synaptics audio drivers installed from being offered Windows 10, version 2004 or Windows Server, version 2004 until the driver has been updated.
