# Prerequisites for Deploying DirectAccess_1
The following table lists the prerequisites necessary for using the configuration wizards to deploy DirectAccess.  
  
|||  
|-|-|  
|Scenario|Prerequisites|  
|[Deploy a Single DirectAccess Server Using the Getting Started Wizard](../Topic/Deploy-a-Single-DirectAccess-Server-Using-the-Getting-Started-Wizard.md)|-   Windows Firewall must be enabled on all profiles<br /><br />-   Only supported for clients running [!INCLUDE[winblue_client_ent_2](../Token/winblue_client_ent_2_md.md)] and [!INCLUDE[win8_client_ent_2](../Token/win8_client_ent_2_md.md)].<br /><br />-   A public key infrastructure is not required.<br /><br />-   Not supported for deploying two\-factor authentication. Domain credentials are required for authentication.<br /><br />-   Automatically deploys DirectAccess to all mobile computers in the current domain.<br /><br />-   Traffic to the Internet does not go through DirectAccess. Force tunnel configuration is not supported.<br /><br />-   DirectAccess server is the network location server.<br /><br />-   Network Access Protection \(NAP\) is not supported.<br /><br />-   Changing policies by using a feature other than the DirectAccess management console or Windows PowerShell cmdlets is not supported.<br /><br />-   For a multisite configuration, now or in the future, first follow the guidance in [Deploy a Single DirectAccess Server with Advanced Settings](../Topic/Deploy-a-Single-DirectAccess-Server-with-Advanced-Settings.md).|  
|[Deploy a Single DirectAccess Server with Advanced Settings](../Topic/Deploy-a-Single-DirectAccess-Server-with-Advanced-Settings.md)|-   A public key infrastructure must be deployed.<br />    For more information, see [Test Lab Guide Mini\-Module: Basic PKI for Windows Server 2012](http://social.technet.microsoft.com/wiki/contents/articles/7862.test-lab-guide-mini-module-basic-pki-for-windows-server-2012.aspx).<br /><br />-   Windows Firewall must be enabled on all profiles.<br /><br /><ul><li>Computers that are running the following operating systems are supported as DirectAccess clients.<br /><br /><ul><li>[!INCLUDE[winblue_server_1](../Token/winblue_server_1_md.md)]</li><li>[!INCLUDE[winblue_client_ent_2](../Token/winblue_client_ent_2_md.md)]</li><li>[!INCLUDE[win8_server_1](../Token/win8_server_1_md.md)]</li><li>[!INCLUDE[win8_client_ent_2](../Token/win8_client_ent_2_md.md)]</li><li>[!INCLUDE[firstref_server_7](../Token/firstref_server_7_md.md)]</li><li>[!INCLUDE[nextref_client_7ult](../Token/nextref_client_7ult_md.md)]</li><li>[!INCLUDE[nextref_client_7ent](../Token/nextref_client_7ent_md.md)]</li></ul></li></ul><br />-   Force tunnel configuration is not supported with KerbProxy authentication.<br /><br />-   Changing policies by using a feature other than the DirectAccess management console or Windows PowerShell cmdlets is not supported.<br /><br />-   Separating NAT64\/DNS64 and IPHTTPS server roles on another server is not supported.|  
|[Deploy Remote Access in a Cluster](../Topic/Deploy-Remote-Access-in-a-Cluster.md)|-   Default load balancing is through the Network Load Balancing \(NLB\) feature in Windows Server.<br /><br />-   External load balancers are supported.<br /><br />-   Unicast mode is the default and recommended mode for NLB.<br /><br />-   Changing policies by using a feature other than the DirectAccess management console or Windows PowerShell cmdlets is not supported.<br /><br />-   When NLB or an external load balancer is used, the IPHTTPS prefix must remain \/59.<br /><br />-   Load balanced nodes must be in the same IPv4 subnet.<br /><br />-   In external load balancer deployments, if remote management is needed, DirectAccess clients cannot use Teredo. Only IPHTTPS can be used for end\-to\-end communication.<br /><br />-   All known hotfixes for Network Load Balancing external load balancing must be installed.|  
|[Deploy Multiple Remote Access Servers in a Multisite Deployment](../Topic/Deploy-Multiple-Remote-Access-Servers-in-a-Multisite-Deployment.md)|-   For a multisite configuration, now or in the future, first follow the guidance in [Deploy a Single DirectAccess Server with Advanced Settings](../Topic/Deploy-a-Single-DirectAccess-Server-with-Advanced-Settings.md).<br /><br />-   Clients running [!INCLUDE[nextref_client_7](../Token/nextref_client_7_md.md)] always connect to a specific site. They cannot connect to the closest site based on the location of the client \(unlike clients running [!INCLUDE[winblue_client_2](../Token/winblue_client_2_md.md)] and [!INCLUDE[win8_client_2](../Token/win8_client_2_md.md)]\).<br /><br />-   Changing policies by using a feature other than the DirectAccess management console or Windows PowerShell cmdlets is not supported.<br /><br />-   A public key infrastructure must be deployed.<br />    For more information, see [Test Lab Guide Mini\-Module: Basic PKI for Windows Server 2012](http://social.technet.microsoft.com/wiki/contents/articles/7862.test-lab-guide-mini-module-basic-pki-for-windows-server-2012.aspx).|  
|[Deploy Remote Access with OTP Authentication_](../Topic/Deploy-Remote-Access-with-OTP-Authentication_.md)|-   Before you deploy a one\-time password authentication, follow the guidance in Deploy a single Remote Access server with advanced settings .<br /><br />-   Clients running [!INCLUDE[nextref_client_7ent](../Token/nextref_client_7ent_md.md)] and [!INCLUDE[nextref_client_7ult](../Token/nextref_client_7ult_md.md)] need to use DCA 2.0 to support one\-time password authentication.<br /><br />-   One\-time password authentication does not support a PIN change.<br /><br />-   A public key infrastructure must be deployed.<br />    For more information, see [Test Lab Guide Mini\-Module: Basic PKI for Windows Server 2012](http://social.technet.microsoft.com/wiki/contents/articles/7862.test-lab-guide-mini-module-basic-pki-for-windows-server-2012.aspx).<br /><br />-   Changing policies by using a feature other than the DirectAccess management console or Windows PowerShell cmdlets is not supported.|  
|[Deploy Remote Access in a Multi-Forest Environment](../Topic/Deploy-Remote-Access-in-a-Multi-Forest-Environment.md)|-   Two\-way trust is required.|  
|[Manage DirectAccess Clients Remotely_1](../Topic/Manage-DirectAccess-Clients-Remotely_1.md)|-   Windows Firewall must be enabled on all profiles.<br /><br /><ul><li>Computers that are running the following operating systems are supported as DirectAccess clients.<br /><br /><ul><li>[!INCLUDE[winblue_server_1](../Token/winblue_server_1_md.md)]</li><li>[!INCLUDE[winblue_client_ent_2](../Token/winblue_client_ent_2_md.md)]</li><li>[!INCLUDE[win8_server_1](../Token/win8_server_1_md.md)]</li><li>[!INCLUDE[win8_client_ent_2](../Token/win8_client_ent_2_md.md)]</li><li>[!INCLUDE[firstref_server_7](../Token/firstref_server_7_md.md)]</li><li>[!INCLUDE[nextref_client_7ult](../Token/nextref_client_7ult_md.md)]</li><li>[!INCLUDE[nextref_client_7ent](../Token/nextref_client_7ent_md.md)]</li></ul></li></ul><br />-   Changing policies by using a feature other than the DirectAccess management console or Windows PowerShell cmdlets is not supported.|  
|[Migrate from Forefront UAG SP1 DirectAccess to Windows Server 2012](../Topic/Migrate-from-Forefront-UAG-SP1-DirectAccess-to-Windows-Server-2012.md)|-   If NAP is used in Forefront Unified Access Gateway, NAP requires a separate Network Policy Server.<br />    NAP was deprecated in Windows Server 2012 R2. This means that NAP may not be supported in future versions of Windows. New deployments with NAP are not recommended.|
